# Configuration pfSense avec assistance IA

## Objectif

Configurer et sécuriser un pare-feu pfSense pour protéger un réseau d'entreprise en exploitant l'IA pour générer des règles de filtrage optimales, détecter des erreurs de configuration et proposer des stratégies de sécurité adaptées au contexte métier.

## Contexte entreprise

Une PME de 50 employés migre d'un routeur basique vers une solution pfSense professionnelle :
- Besoin de segmentation réseau (VLAN Admin, Users, Serveurs, Invités)
- VPN site-to-site avec agence distante
- Filtrage web (blocage sites malveillants, catégories interdites)
- Monitoring trafic et détection d'anomalies
- Haute disponibilité (HA) souhaitée

**Infrastructure** :
- 2x serveurs pfSense (HA avec CARP)
- WAN : Fibre 1 Gbps
- LAN : 4 VLANs (10, 20, 30, 40)
- 5 serveurs internes (web, mail, fichiers)

**Objectif** : Configurer pfSense en 3 jours au lieu de 10 jours.

## Problème

Configurer pfSense de zéro implique :
- Comprendre l'architecture des règles (ordre, priorité)
- Définir une politique de sécurité cohérente
- Configurer NAT, port forwarding complexe
- Implémenter VPN OpenVPN ou IPSec
- Optimiser les performances firewall
- Activer logging et monitoring

**Temps estimé** : 8-10 jours pour une configuration complète

**Difficultés** :
- Interface web riche mais complexe
- Ordre des règles firewall critique
- Multiples options de sécurité
- Configuration VPN technique

## Solution traditionnelle

### Méthode manuelle

```
1. Lire documentation pfSense (500+ pages)
2. Définir manuellement les règles firewall
3. Tester chaque règle individuellement
4. Corriger les erreurs une par une
5. Consulter forums pfSense pour problèmes
```

**Problèmes** :
- Courbe d'apprentissage longue
- Risque de règles incohérentes
- Temps de test/debug important
- Documentation dense

## Apport de l'IA

L'IA permet de :
- Générer des règles firewall optimales basées sur le contexte
- Analyser la logique de règles existantes
- Détecter les erreurs de configuration (règles inutiles, conflits)
- Proposer une architecture de sécurité complète
- Créer des configurations VPN étape par étape
- Expliquer les concepts complexes en langage clair

**Temps avec IA** : 2-3 jours pour une configuration complète

**Gain de productivité** : 70-75%

### Étape 1 : Définir l'architecture avec l'IA

**Prompt** :

```
Je configure un pare-feu pfSense pour une PME de 50 employés.

Architecture réseau :
- WAN : Fibre 1 Gbps (IP publique)
- LAN segmenté en 4 VLANs :
  * VLAN 10 : Administration (10.0.10.0/24) - 5 postes IT
  * VLAN 20 : Utilisateurs (10.0.20.0/24) - 50 postes
  * VLAN 30 : Serveurs (10.0.30.0/24) - 5 serveurs (web, mail, fichiers)
  * VLAN 40 : Invités (10.0.40.0/24) - WiFi invités

Besoins :
1. Segmentation stricte entre VLANs
2. VLAN 20 accès Internet + serveurs internes
3. VLAN 40 accès Internet uniquement (isolé)
4. VLAN 30 accessible depuis VLAN 10 seulement
5. Serveur web (10.0.30.10) accessible depuis Internet (port 443)

Propose une architecture de règles firewall pfSense optimale.

Format : Liste des règles par interface avec explications
```

**Réponse IA** :

```
=== ARCHITECTURE FIREWALL PFSENSE ===

PRINCIPE : pfSense applique les règles dans l'ordre (top-down) et s'arrête à la première correspondance.
Stratégie : "Default Deny" (bloquer tout sauf ce qui est explicitement autorisé)

=== RÈGLES WAN ===
Interface : WAN
Stratégie : Bloquer tout sauf services exposés

1. [ALLOW] Protocol: TCP, Source: any, Port: any → Destination: IP_PUBLIQUE, Port: 443
   → Autoriser HTTPS vers serveur web interne (NAT+Port Forward)

2. [ALLOW] Protocol: UDP, Source: any, Port: any → Destination: WAN_address, Port: 1194
   → Autoriser OpenVPN (si VPN configuré)

3. [BLOCK] Protocol: any, Source: any → Destination: any
   → Bloquer tout le reste (règle implicite mais explicite pour clarté)

=== RÈGLES LAN VLAN 10 (ADMIN) ===
Interface : VLAN10
Stratégie : Accès complet (administrateurs)

1. [ALLOW] Protocol: any, Source: VLAN10_net → Destination: any
   → Les admins ont accès à tout (VLAN + Internet)

=== RÈGLES LAN VLAN 20 (USERS) ===
Interface : VLAN20
Stratégie : Accès Internet + serveurs internes uniquement

1. [ALLOW] Protocol: TCP, Source: VLAN20_net → Destination: 10.0.30.10, Port: 443
   → Accès serveur web interne

2. [ALLOW] Protocol: TCP, Source: VLAN20_net → Destination: 10.0.30.11, Ports: 587,993
   → Accès serveur mail (SMTP+IMAP)

3. [ALLOW] Protocol: TCP/UDP, Source: VLAN20_net → Destination: 10.0.30.12, Ports: 445,139
   → Accès serveur fichiers (SMB)

4. [ALLOW] Protocol: TCP/UDP, Source: VLAN20_net, Port: any → Destination: !RFC1918, Port: any
   → Accès Internet (destination NON privée)

5. [BLOCK] Protocol: any, Source: VLAN20_net → Destination: RFC1918
   → Bloquer accès aux autres VLANs

6. [BLOCK] Protocol: any, Source: VLAN20_net → Destination: any
   → Bloquer tout le reste

=== RÈGLES LAN VLAN 30 (SERVEURS) ===
Interface : VLAN30
Stratégie : Accès sortant limité, entrant depuis VLAN10 seulement

1. [ALLOW] Protocol: TCP/UDP, Source: VLAN30_net → Destination: !RFC1918, Ports: 80,443
   → Serveurs peuvent mettre à jour via Internet (HTTP/HTTPS)

2. [ALLOW] Protocol: UDP, Source: VLAN30_net → Destination: 10.0.10.5, Port: 53
   → DNS vers serveur interne

3. [BLOCK] Protocol: any, Source: VLAN30_net → Destination: any
   → Bloquer tout le reste

=== RÈGLES LAN VLAN 40 (INVITÉS) ===
Interface : VLAN40
Stratégie : Internet uniquement, isolation totale du LAN

1. [ALLOW] Protocol: TCP/UDP, Source: VLAN40_net → Destination: !RFC1918, Port: any
   → Accès Internet uniquement

2. [BLOCK] Protocol: any, Source: VLAN40_net → Destination: RFC1918
   → Bloquer accès réseau interne

3. [BLOCK] Protocol: any, Source: VLAN40_net → Destination: any
   → Bloquer tout le reste

=== NAT / PORT FORWARDING ===

1. WAN → Serveur Web interne
   Interface: WAN
   Protocol: TCP
   Destination: WAN_address
   Destination Port: 443
   Redirect Target IP: 10.0.30.10
   Redirect Target Port: 443
   Description: HTTPS vers serveur web

=== ORDRE D'APPLICATION ===
1. pfSense traite les règles interface par interface
2. Pour chaque interface : ordre top-down
3. Première correspondance = action appliquée (stop)
4. Si aucune correspondance = BLOCK implicite
```

## Scripts / Commandes / Configurations

### Backup automatique configuration

```bash
#!/bin/bash
# pfsense-backup.sh
# Backup automatique config pfSense via SCP

PFSENSE_IP="10.0.0.1"
BACKUP_DIR="/backups/pfsense"
DATE=$(date +%Y%m%d-%H%M)

mkdir -p "$BACKUP_DIR"

# Télécharger config.xml
scp admin@$PFSENSE_IP:/cf/conf/config.xml "$BACKUP_DIR/config-$DATE.xml"

# Garder 30 derniers backups
ls -t "$BACKUP_DIR"/config-*.xml | tail -n +31 | xargs rm -f

echo "Backup pfSense sauvegardé : $BACKUP_DIR/config-$DATE.xml"
```

### Export règles firewall en CSV

```php
/* Depuis pfSense Shell */
/* System > Advanced > Command Prompt > Execute Shell Command */

<?php
require_once("config.inc");
require_once("filter.inc");

$rules = $config['filter']['rule'];
$csv = "Interface,Action,Protocol,Source,Destination,Port,Description\n";

foreach ($rules as $rule) {
    $csv .= sprintf("%s,%s,%s,%s,%s,%s,%s\n",
        $rule['interface'],
        $rule['type'],
        $rule['protocol'],
        $rule['source']['address'] ?? 'any',
        $rule['destination']['address'] ?? 'any',
        $rule['destination']['port'] ?? 'any',
        $rule['descr']
    );
}

file_put_contents("/tmp/firewall-rules.csv", $csv);
echo "Export CSV créé : /tmp/firewall-rules.csv";
?>
```

## Risques et limites

### Risques de configuration

**Blocage accidentel** : Une règle firewall mal placée peut bloquer tout le trafic. Toujours tester sur VLAN test d'abord.

**Ordre des règles** : pfSense applique top-down. Une règle ALLOW trop large en haut annule les règles restrictives en dessous.

**VPN site-to-site** : Une erreur dans les Phase 1/2 IPsec empêche la connexion. Vérifier les logs IPsec pour debug.

### Limites de l'IA

**Contexte métier spécifique** : L'IA génère des configurations génériques. Adapter aux besoins précis de l'entreprise.

**Versions pfSense** : Certaines options changent entre versions (CE vs Plus). Vérifier compatibilité.

**Performance** : L'IA ne peut pas prédire l'impact performance. Tester en charge réelle.

### Risques opérationnels

**Perte de connectivité** : Modifications firewall peuvent couper l'accès. Toujours avoir un accès console ou IPMI de secours.

**HA mal configurée** : Une mauvaise config CARP peut causer une perte de service. Tester le failover.

## Bonnes pratiques

### Architecture firewall

1. Toujours partir d'une stratégie "Default Deny"
2. Placer les règles les plus spécifiques en haut
3. Documenter chaque règle avec une description claire
4. Regrouper les règles par fonction (VPN, NAT, Inter-VLAN)

### Sécurité

1. Activer logging sur les règles critiques
2. Désactiver les services inutiles (UPnP, SNMP v1/v2)
3. Utiliser des alias pour faciliter la maintenance
4. Mettre à jour pfSense régulièrement
5. Activer pfBlockerNG pour filtrage web/IP

### Haute disponibilité

1. Configurer CARP pour IP virtuelles
2. Synchroniser config entre les 2 pfSense (XMLRPC)
3. Tester le failover régulièrement
4. Monitoring avec alertes sur perte de sync

### Maintenance

1. Sauvegarder la config avant chaque modification
2. Tenir un changelog des règles ajoutées/supprimées
3. Auditer les règles tous les 6 mois (supprimer inutiles)
4. Exporter les règles en CSV pour documentation
